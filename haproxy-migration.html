HAProxy to NGINX Ingress Migration Tool

Migrating from HAProxy? The HAProxy Kubernetes Ingress Controller is actively maintained — this guide is for teams standardizing on the NGINX data plane. It translates HAProxy's annotations, ConfigMap, and Global/Defaults/Backend/TCP CRDs into the F5 NGINX Ingress Controller's ConfigMap + annotation + VirtualServer/Policy model, resource by resource.
Version Reference: This tool is based on HAProxy Kubernetes Ingress Controller v3.2.12 and F5 NGINX Ingress Controller v5.5.1.

Overview

This interactive guide helps you migrate from the HAProxy Kubernetes Ingress Controller (haproxytech/kubernetes-ingress) to the F5 NGINX Ingress Controller (nginx/kubernetes-ingress). It covers the haproxy.org/ annotations at all three HAProxy scopes (ConfigMap, Ingress, and Service — note that NIC has no Service-level annotation tier), the controller ConfigMap, both CRD API groups (ingress.v1.haproxy.org and ingress.v3.haproxy.org: Global, Defaults, Backend, TCP in both, plus Frontend and ValidationRules in v3 only), the legacy tcp-services ConfigMap, and route-acl canary routing — across three tabs:

  • Getting Started — Why teams migrate, NIC CRD overviews, installation, and a migration checklist
  • HAProxy Config Analyzer — Paste annotated Ingress/Service manifests, the HAProxy ConfigMap, or Global/Defaults/Backend/TCP CRs and get automatic migration suggestions with ready-to-use YAML
  • Reference Guide — Complete annotation, CRD, ConfigMap, and controller-flag mapping tables with side-by-side examples

Why Migrate?

The HAProxy Kubernetes Ingress Controller is actively maintained, so moving off it is a choice rather than a forced migration. Teams that migrate usually do it to consolidate:

  • Standardize on the NGINX data plane — One proxy technology across VMs, containers, and Kubernetes means one config language to debug and consistent performance characteristics everywhere
  • Schema-validated CRDs over raw config — HAProxy's advanced features often reach for *-config-snippet escape hatches that inject raw HAProxy configuration; NIC's VirtualServer, Policy, and TransportServer CRDs are type-checked and validated by the API server
  • NGINX Plus upgrade path — F5 NGINX App Protect WAF and DoS, JWT, OIDC, active health checks, and zone-sync are available as NGINX Plus policies with F5 commercial support — the analogue of HAProxy Enterprise's WAF, Global Profiling Engine, and Bot Management
  • Commercial support from F5 — Enterprise support contracts cover the controller and the NGINX data plane together

The two controllers structure configuration differently, and knowing the mental model makes the mappings obvious:

  • HAProxy resolves haproxy.org/ annotations across four tiers (default < ConfigMap < Ingress < Service, highest wins), and its Global/Defaults/Backend/Frontend CRDs replace annotations when both are present
  • F5 NGINX Ingress Controller keeps global defaults in one nginx-config ConfigMap that per-resource annotations and CRDs override — there is no Service-level annotation tier, so Service-scoped HAProxy annotations move to per-Ingress annotations or the ConfigMap
Fair warning: a few HAProxy behaviors differ on NIC — snippet content is a different configuration language (never copyable), redirect and rate-limit reject codes default differently (emit them explicitly), and there is no Service-level annotation tier. The Key Differences table covers these head-on.

Phased Migration Strategy

Rather than performing a "big bang" cutover, F5 recommends a phased migration that validates each application before decommissioning HAProxy. Because HAProxy's CRD groups (ingress.v1.haproxy.org / ingress.v3.haproxy.org) and NIC's CRDs (k8s.nginx.org) are entirely separate API groups, both controllers coexist in one cluster without touching each other's resources — the migration is additive translation, not in-place mutation.

  1. Deploy side-by-side — Install the F5 NGINX Ingress Controller alongside HAProxy. Existing HAProxy-annotated resources are untouched; plain Ingress resources stay with HAProxy until you flip their ingressClassName.
  2. Translate service-by-service — Create the VirtualServer, Policy, and TransportServer resources for one application at a time (the Config Analyzer generates first drafts). Validate against the NIC LoadBalancer IP with curl --resolve app.example.com:443:<NIC-IP>, then flip that host's DNS record. Plain-Ingress users instead switch ingressClassName from the HAProxy class to nginx after converting annotations.
  3. Decommission — After every host has been cut over and validated, remove the HAProxy Deployment, its IngressClass, both *.haproxy.org CRD groups, and the CRs they managed.
Tip: Migrate certificates first. NIC has no on-the-fly certificate signing (HAProxy's generate-certificates has no equivalent) — install cert-manager and issue certificates for each host before the DNS cutover, or pre-provision kubernetes.io/tls Secrets, so the new controller serves valid TLS from the first request.

CRDs Summary

The F5 NGINX Ingress Controller uses Custom Resource Definitions (CRDs) to extend Kubernetes with NGINX-specific configuration. CRDs provide type-safe, validated configuration that goes beyond what standard Ingress annotations can offer.

VirtualServer CRD

The primary CRD for configuring HTTP/HTTPS load balancing. It replaces Ingress resources and provides advanced traffic management capabilities with full validation.

  • Traffic Splitting: Route percentages of traffic to different backends for canary/blue-green deployments
  • Content-Based Routing: Route based on headers, cookies, arguments, or variables
  • Path Rewriting: Modify request URIs with regex support before forwarding to backends
  • Error Pages: Custom responses for specific HTTP error codes
  • Request/Response Headers: Add, modify, or remove headers in transit
  • Health Checks: Active health monitoring of upstream servers
  • Session Persistence: Cookie-based sticky sessions

VirtualServerRoute CRD

Enables delegation of route configuration to different namespaces. Allows teams to manage their own routing rules while maintaining centralized control over the domain.

  • Cross-Namespace Routing: Reference services in different namespaces
  • Path Delegation: Delegate path prefixes to other teams/namespaces
  • Independent Updates: Teams update routes without modifying the parent VirtualServer
  • Access Control: Limit which namespaces can define routes for a host

Policy CRD

Reusable security and traffic policies that can be attached to VirtualServers or specific routes. Policies enable consistent security controls across multiple applications.

  • Rate Limiting: Limit request rates by IP, header, or custom key
  • Access Control: IP allowlists and denylists
  • Basic Auth: Username/password authentication via htpasswd
  • JWT Validation: Validate JSON Web Tokens (Plus)
  • OIDC: Native OpenID Connect integration (Plus)
  • API Key Auth: Authenticate via API keys in headers/query
  • mTLS: Client certificate verification (ingress) and backend TLS (egress)
  • WAF: F5 WAF for NGINX web application firewall (Plus)

TransportServer CRD

Configures TCP/UDP load balancing for non-HTTP protocols. Essential for databases, message queues, gaming servers, and other Layer 4 applications.

  • TCP Load Balancing: MySQL, PostgreSQL, Redis, custom TCP services
  • UDP Load Balancing: DNS, RADIUS, gaming protocols
  • TLS Passthrough: Pass encrypted traffic directly to backends
  • Health Checks: TCP/UDP health monitoring
  • Session Persistence: IP hash-based session affinity

GlobalConfiguration CRD

Cluster-wide configuration for the F5 NGINX Ingress Controller. Defines global listeners for TCP/UDP and TLS passthrough that TransportServers can reference.

  • TCP/UDP Listeners: Define ports for non-HTTP traffic
  • TLS Passthrough Listeners: Configure SNI-based routing
  • Global Settings: Controller-wide configuration options

Installing CRDs

CRDs can be installed via Helm or manually using manifests, depending on your deployment method. See the F5 NGINX Ingress Controller installation docs for full instructions.

Helm (example)

If you install the F5 NGINX Ingress Controller via Helm, CRDs are included automatically. To ensure they are enabled:

helm install nginx-ingress oci://ghcr.io/nginx/charts/nginx-ingress --version 2.6.1 --set controller.enableCustomResources=true

The controller.enableCustomResources flag is true by default. If you previously disabled it, re-enable it to use CRD resources. See the Helm installation guide for details.

Manifests (example)

If you deploy using manifests, install CRDs manually before deploying the controller:

kubectl apply -f https://raw.githubusercontent.com/nginx/kubernetes-ingress/v5.5.1/deploy/crds.yaml

Replace the version number with your target controller version. See the manifest installation guide for the full step-by-step process.

Note: The crds.yaml file contains all CRDs (VirtualServer, VirtualServerRoute, Policy, TransportServer, GlobalConfiguration) in a single manifest.

Migration Checklist

Progress is saved in your browser.
  • Install NIC CRDs via Helm (controller.enableCustomResources=true) or manifests (deploy/crds.yaml)
  • Inventory haproxy.org/ annotations at all three scopes — ConfigMap, Ingress, and Service objects (NIC has no Service-level annotation tier)
  • Catalog the HAProxy controller ConfigMap keys and any cr-* pointers (cr-global/cr-defaults/cr-backend/cr-frontend-*)
  • Inventory both CRD groups (ingress.v1.haproxy.org & ingress.v3.haproxy.org: Global/Defaults/Backend/TCP in both, Frontend/ValidationRules v3-only) and drop the cr-* pointer indirection — migrate the referenced CR's content, not the pointer
  • Convert auth-secret Secrets to type nginx.org/htpasswd (concatenate the per-user entries into one htpasswd file under key htpasswd)
  • Re-type client-ca/server-ca Secrets to nginx.org/ca — move the CA from tls.crt to ca.crt
  • Convert allow-list/deny-list (and the deprecated whitelist/blacklist aliases) to accessControl Policies — allow XOR deny, so a rule needing both becomes two Policies
  • Convert rate-limit-* to a rateLimit Policy (or nginx.org/limit-req-*) and emit rejectCode: 403 explicitly (NIC default is 503)
  • Convert ssl-redirect/request-redirect and emit the redirect code 302 explicitly (NIC http-redirect-code/ActionRedirect default is 301)
  • Re-express route-acl canary as VirtualServer splits[] (from rand()) and matches[].conditions[] (cookie/header/argument); src/CIDR ACLs become accessControl Policies
  • Convert cookie-persistence to VirtualServer upstream sessionCookie (omit expires)
  • Collapse the cors-* annotations into one cors Policy attached via nginx.org/policies — emit maxAge explicitly and enumerate allowMethods (no *)
  • Convert the tcp-services ConfigMap and TCP CRs to GlobalConfiguration listeners + TransportServers — add an SNI spec.host for :ssl (TLS-terminated) entries
  • Convert ssl-passthrough to a TransportServer (tls-passthrough listener) and set the -enable-tls-passthrough flag
  • Re-author *-config-snippet content as NGINX directives (main/http/server/location context) and set -enable-snippets for resource snippets
  • Relocate Service-scoped annotations to per-Ingress annotations or the ConfigMap (NIC has no Service tier)
  • Split timeout-server into both nginx.org/proxy-read-timeout and nginx.org/proxy-send-timeout
  • Review pathType: Prefix boundary semantics — plain Ingress renders Prefix as a raw prefix (use a VirtualServer regex route to preserve path-segment boundaries)
  • Hand-translate log-format/log-format-tcp — HAProxy %-tokens become nginx $-variables (never copy verbatim)
  • Re-author ModSecurity/CRS rules as an App Protect waf Policy (NGINX Plus) — HAProxy Enterprise WAF rules are not portable
  • Rebuild Prometheus/Grafana dashboards — NIC metric names (nginx_ingress_*) differ from HAProxy's on a dedicated :9113 port
  • Validate in a staging environment before cutover
  • Cut over DNS host-by-host (validate with curl --resolve app.example.com:443:<NIC-IP>), then decommission the HAProxy Deployment, IngressClass, and CRDs

  name: my-app
  annotations:
    haproxy.org/allow-list: "10.0.0.0/8,192.168.1.0/24"

F5 NGINX Ingress Controller
apiVersion: k8s.nginx.org/v1
kind: Policy
metadata:
  name: access-control-allow
spec:
  accessControl:
    allow:
      - 10.0.0.0/8
      - 192.168.1.0/24

# Attach via VirtualServer spec.policies, or on Ingress via:
#   annotations:
#     nginx.org/policies: "access-control-allow"
# accessControl matches on $remote_addr — behind another proxy, also set the
# real-ip ConfigMap keys (set-real-ip-from, real-ip-header, real-ip-recursive).

                            

                            
allow and deny are mutually exclusive: a Policy's accessControl must set exactly one of allow or deny — an Ingress that carries both haproxy.org/allow-list and haproxy.org/deny-list becomes two Policies, both listed in nginx.org/policies. allow is a whitelist (everything else gets 403); deny blocks the listed ranges and permits the rest.

                            
Note: Policy CRDs attach to Ingress resources via nginx.org/policies since v5.4.0 (requires the controller flag -enable-custom-resources), or to VirtualServer/VirtualServerRoute spec.policies. accessControl matches on $remote_addr — behind another proxy, set the real-ip ConfigMap keys (set-real-ip-from, real-ip-header, real-ip-recursive) so the real client IP is evaluated. When a value starts with patterns/ it references a ConfigMap pattern file — inline those CIDRs into the Policy by hand. On mergeable Ingresses the annotation must sit on the master.

#

#     proxy:
#       upstream: backend

Traffic Splitting

            
Key difference: HAProxy's route-acl injects a raw use_backend rule (first match wins, config order) with no weight field — canary is expressed purely as an ACL. The F5 NGINX Ingress Controller uses declarative VirtualServer splits[] (percentages) or matches[] (conditions). Only a small set of portable ACL primitives convert.

                            haproxy.org/route-acl

# not the Ingress.

---

                port:

      service: canary-svc

TCP Services

            
L4 model: HAProxy's TCP CRD and the legacy tcp-services ConfigMap both become a GlobalConfiguration TCP listener plus a TransportServer per entry. The listener port must be pre-declared in the single GlobalConfiguration (-global-configuration <ns>/<name>) and exposed on the controller Service. L4 content-switching/ACL rules do not port (streamSnippets only).
