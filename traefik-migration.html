Traefik to NGINX Ingress Migration Tool

Migrating from Traefik? Traefik Proxy is an actively maintained project — this guide is for teams standardizing on the NGINX data plane. It translates Traefik's IngressRoute + Middleware model into the F5 NGINX Ingress Controller's VirtualServer + Policy model, resource by resource.
Version Reference: This tool is based on Traefik Proxy v3.7.6 and F5 NGINX Ingress Controller v5.5.1.

Overview

This interactive guide helps you migrate from Traefik Proxy (traefik/traefik) to the F5 NGINX Ingress Controller (nginx/kubernetes-ingress). It covers all 24 OSS HTTP middlewares, the traefik.ingress.kubernetes.io/ Ingress annotations, Traefik's Kubernetes CRDs (IngressRoute, TraefikService, TLSOption, TLSStore, ServersTransport, and the TCP/UDP variants), and static-configuration translation across three tabs:

  • Getting Started — Why teams migrate, NIC CRD overviews, installation, and a migration checklist
  • Traefik Config Analyzer — Paste IngressRoute, Middleware, or annotated Ingress manifests and get automatic migration suggestions with ready-to-use YAML
  • Reference Guide — Complete middleware, annotation, and CRD mapping tables with side-by-side examples

Why Migrate?

Unlike migrations forced by a project sunset, moving off Traefik is a choice — Traefik Proxy is actively maintained. Teams that migrate usually do it to consolidate:

  • Standardize on the NGINX data plane — One proxy technology across VMs, containers, and Kubernetes means one set of operational knowledge, one config language to debug, and consistent performance characteristics everywhere
  • Battle-tested NGINX core — The F5 NGINX Ingress Controller configures stock NGINX (OSS or Plus), so decades of NGINX tuning knowledge, modules, and monitoring practices apply directly
  • Flatter resource graph — An IngressRoute plus a chain of Middleware objects becomes one VirtualServer plus reusable Policies: schema-validated CRDs with fewer cross-resource references to trace
  • NGINX Plus upgrade path — JWT, OIDC, WAF, and active health checks are available as NGINX Plus policies without switching to a different commercial product family (API-key auth maps to the NIC apiKey Policy, which runs on NGINX OSS)
  • Commercial support from F5 — Enterprise support contracts cover the controller and the data plane together

The two controllers structure configuration differently, and knowing the mental model makes the mappings obvious:

  • Traefik attaches chainable Middleware objects to routers, terminates TLS with a built-in ACME client, and splits configuration between static entryPoints and dynamic providers
  • F5 NGINX Ingress Controller attaches reusable Policy objects to VirtualServers (or Ingress resources via nginx.org/policies), integrates with cert-manager for certificates, and keeps global settings in one ConfigMap
Fair warning: a few Traefik conveniences work differently on NIC — the built-in Let's Encrypt client is replaced by cert-manager, the dashboard by Prometheus/Grafana, and per-router observability toggles by global settings. The Key Differences table covers these head-on.

Phased Migration Strategy

Rather than performing a "big bang" cutover, F5 recommends a phased migration that validates each application before decommissioning Traefik. Because Traefik's CRDs (traefik.io) and NIC's CRDs (k8s.nginx.org) are separate API groups, both controllers coexist in one cluster without touching each other's resources — the migration is additive translation, not in-place mutation.

  1. Deploy side-by-side — Install the F5 NGINX Ingress Controller alongside Traefik. Existing IngressRoutes are untouched; plain Ingress resources stay with Traefik until you flip their ingressClassName.
  2. Translate service-by-service — Create the VirtualServer, Policy, and TransportServer resources for one application at a time (the Config Analyzer generates first drafts). Validate against the NIC LoadBalancer IP with curl --resolve app.example.com:443:<NIC-IP>, then flip that host's DNS record. Plain-Ingress users instead switch ingressClassName from traefik to nginx after converting annotations.
  3. Decommission — After every host has been cut over and validated, remove the Traefik Deployment, its IngressClass, the traefik.io CRDs, and the Middleware/TraefikService objects they managed.
Tip: Migrate certificates first. NIC has no built-in ACME client — install cert-manager and issue certificates for each host before the DNS cutover so the new controller serves valid TLS from the first request (Traefik's acme.json certificates are not Kubernetes Secrets and cannot be reused directly).

CRDs Summary

The F5 NGINX Ingress Controller uses Custom Resource Definitions (CRDs) to extend Kubernetes with NGINX-specific configuration. CRDs provide type-safe, validated configuration that goes beyond what standard Ingress annotations can offer.

VirtualServer CRD

The primary CRD for configuring HTTP/HTTPS load balancing. It replaces Ingress resources and provides advanced traffic management capabilities with full validation.

  • Traffic Splitting: Route percentages of traffic to different backends for canary/blue-green deployments
  • Content-Based Routing: Route based on headers, cookies, arguments, or variables
  • Path Rewriting: Modify request URIs with regex support before forwarding to backends
  • Error Pages: Custom responses for specific HTTP error codes
  • Request/Response Headers: Add, modify, or remove headers in transit
  • Health Checks: Active health monitoring of upstream servers
  • Session Persistence: Cookie-based sticky sessions

VirtualServerRoute CRD

Enables delegation of route configuration to different namespaces. Allows teams to manage their own routing rules while maintaining centralized control over the domain.

  • Cross-Namespace Routing: Reference services in different namespaces
  • Path Delegation: Delegate path prefixes to other teams/namespaces
  • Independent Updates: Teams update routes without modifying the parent VirtualServer
  • Access Control: Limit which namespaces can define routes for a host

Policy CRD

Reusable security and traffic policies that can be attached to VirtualServers or specific routes. Policies enable consistent security controls across multiple applications.

  • Rate Limiting: Limit request rates by IP, header, or custom key
  • Access Control: IP allowlists and denylists
  • Basic Auth: Username/password authentication via htpasswd
  • JWT Validation: Validate JSON Web Tokens (Plus)
  • OIDC: Native OpenID Connect integration (Plus)
  • API Key Auth: Authenticate via API keys in headers/query
  • mTLS: Client certificate verification (ingress) and backend TLS (egress)
  • WAF: F5 WAF for NGINX web application firewall (Plus)

TransportServer CRD

Configures TCP/UDP load balancing for non-HTTP protocols. Essential for databases, message queues, gaming servers, and other Layer 4 applications.

  • TCP Load Balancing: MySQL, PostgreSQL, Redis, custom TCP services
  • UDP Load Balancing: DNS, RADIUS, gaming protocols
  • TLS Passthrough: Pass encrypted traffic directly to backends
  • Health Checks: TCP/UDP health monitoring
  • Session Persistence: IP hash-based session affinity

GlobalConfiguration CRD

Cluster-wide configuration for the F5 NGINX Ingress Controller. Defines global listeners for TCP/UDP and TLS passthrough that TransportServers can reference.

  • TCP/UDP Listeners: Define ports for non-HTTP traffic
  • TLS Passthrough Listeners: Configure SNI-based routing
  • Global Settings: Controller-wide configuration options

Installing CRDs

CRDs can be installed via Helm or manually using manifests, depending on your deployment method. See the F5 NGINX Ingress Controller installation docs for full instructions.

Helm (example)

If you install the F5 NGINX Ingress Controller via Helm, CRDs are included automatically. To ensure they are enabled:

helm install nginx-ingress oci://ghcr.io/nginx/charts/nginx-ingress --version 2.6.1 --set controller.enableCustomResources=true

The controller.enableCustomResources flag is true by default. If you previously disabled it, re-enable it to use CRD resources. See the Helm installation guide for details.

Manifests (example)

If you deploy using manifests, install CRDs manually before deploying the controller:

kubectl apply -f https://raw.githubusercontent.com/nginx/kubernetes-ingress/v5.5.1/deploy/crds.yaml

Replace the version number with your target controller version. See the manifest installation guide for the full step-by-step process.

Note: The crds.yaml file contains all CRDs (VirtualServer, VirtualServerRoute, Policy, TransportServer, GlobalConfiguration) in a single manifest.

Migration Checklist

Progress is saved in your browser.
  • Install NIC CRDs via Helm (controller.enableCustomResources=true) or manifests (deploy/crds.yaml)
  • Inventory Traefik resources cluster-wide: kubectl get ingressroutes,ingressroutetcps,ingressrouteudps,middlewares,middlewaretcps,traefikservices,tlsoptions,tlsstores,serverstransports,serverstransporttcps -A
  • Audit entryPoints and map custom ports to GlobalConfiguration listeners or nginx.org/listen-ports
  • Replace Let's Encrypt certResolvers with cert-manager Issuers (or pre-provisioned Secrets) before cutover — acme.json certificates cannot be reused
  • Convert IngressRoute match rules to VirtualServer host/routes[].path/matches
  • Convert basicAuth middlewares to the basicAuth Policy — re-create Secrets as type nginx.org/htpasswd (key htpasswd)
  • Convert forwardAuth middlewares to the externalAuth Policy (v5.5.0+)
  • Convert ipAllowList middlewares to the accessControl Policy (configure set-real-ip-from when behind a proxy)
  • Convert rateLimit middlewares to the rateLimit Policy — convert average/period to r/s or r/m and review burst semantics
  • Convert headers middlewares — CORS to the cors Policy, HSTS to nginx.org/hsts*, custom headers to VirtualServer requestHeaders/responseHeaders
  • Convert stripPrefix/addPrefix/replacePath chains to VirtualServer action.proxy.rewritePath or nginx.org/rewrites
  • Convert errors middlewares to VirtualServer errorPages (explicit status codes — no ranges)
  • Convert redirectScheme/redirectRegex to nginx.org/redirect-to-https or VirtualServer action.redirect
  • Convert weighted TraefikServices to VirtualServer splits; mirroring is snippet-based (no native field — and percent sampling has no equivalent)
  • Convert sticky sessions (sticky.cookie) to VirtualServer upstream sessionCookie
  • Convert IngressRouteTCP/UDP and TLS passthrough (HostSNI) to TransportServer + GlobalConfiguration listeners
  • Convert TLSOptions to ConfigMap ssl-protocols/ssl-ciphers; move clientAuth to ingressMTLS policies
  • Convert ServersTransports to egressMTLS policies and VirtualServer upstream TLS/timeout settings
  • Audit Traefik Hub middlewares (JWT, OIDC, WAF) and map them to NGINX Plus policies; the Hub API-key middleware maps to the NIC apiKey Policy, which runs on NGINX OSS
  • Recreate observability — Prometheus scrape target (-enable-prometheus-metrics), log-format, OTel ConfigMap keys; replace Traefik dashboard views with Grafana
  • Test in staging environment
  • Cut over DNS host-by-host, update monitoring and alerting, then decommission the Traefik Deployment, IngressClass, and CRDs

Caveats: ipStrategy has no per-route equivalent — the client IP that accessControl evaluates is determined globally via the ConfigMap keys set-real-ip-from (trusted proxy CIDRs), real-ip-header (for example X-Forwarded-For), and real-ip-recursive (the closest match to depth/excludedIPs/ipv6Subnet resolution). rejectStatusCode is not configurable — accessControl always rejects with 403 (Traefik's default).

                            Middleware passTLSClientCert
                            VirtualServer CRD action.proxy.requestHeaders.set with ${ssl_client_escaped_cert} / ${ssl_client_s_dn} + Policy CRD ingressMTLS
                        

                        

                            

                                
Traefik Proxy
apiVersion: traefik.io/v1alpha1
kind: Middleware
metadata:
  name: pass-client-cert
spec:
  passTLSClientCert:
    pem: true
    info:
      subject:
        commonName: true

F5 NGINX Ingress Controller
# 1. Forward the client certificate via request headers
apiVersion: k8s.nginx.org/v1
kind: VirtualServer
metadata:
  name: my-app
spec:
  host: app.example.com
  tls:
    secret: app-tls-secret  # ingressMTLS requires TLS termination
  policies:
    - name: ingress-mtls-policy  # requests + verifies the client cert
  upstreams:
    - name: backend
      service: app-service
      port: 80
  routes:
    - path: /
      action:
        proxy:
          upstream: backend
          requestHeaders:
            set:
              - name: X-Forwarded-Tls-Client-Cert  # pem: true
                value: ${ssl_client_escaped_cert}
              - name: X-Forwarded-Tls-Client-Cert-Info  # info.subject.*
                value: ${ssl_client_s_dn}
              - name: X-Forwarded-Tls-Client-Cert-Issuer  # info.issuer.*
                value: ${ssl_client_i_dn}
---
# 2. Without an ingressMTLS Policy, NGINX never requests the
#    client certificate and both variables stay empty
apiVersion: k8s.nginx.org/v1
kind: Policy
metadata:
  name: ingress-mtls-policy
spec:
  ingressMTLS:
    clientCertSecret: ingress-mtls-secret  # Secret type nginx.org/ca (CA under ca.crt)
    verifyClient: "on"

                            

                            
Note: Traefik's single X-Forwarded-Tls-Client-Cert-Info header concatenates both the Subject and Issuer DNs; in NIC these map to ${ssl_client_s_dn} (subject) and ${ssl_client_i_dn} (issuer) respectively. Individual sub-fields (commonName, organization, etc.) remain DN components rather than one-to-one headers. In Traefik, the entry point's TLSOption clientAuth requests the certificate that passTLSClientCert forwards; here the ingressMTLS Policy plays that role (clientAuthType maps to verifyClient), and it must be referenced at VirtualServer spec.policies level, not on a route.

On Ingress resources use the nginx.org/basic-auth-realm and nginx.org/basic-auth-secret annotations instead (see the Annotation Approach). Only the first basicAuth Policy reference is applied; subsequent references are ignored. headerField and removeHeader have no equivalent — the authenticated user is not forwarded to the backend and the Authorization header passes through unchanged.

Behavior differences: All client request headers are always forwarded to the auth service — authRequestHeaders has no equivalent (restrict headers only via authSnippets). The request body is never forwarded — forwardBody/maxBodySize have no equivalent. Auth-server response headers are not copied to the backend request — replicate authResponseHeaders/authResponseHeadersRegex with authSnippets (auth_request_set). addAuthCookiesToResponse has no equivalent — the controller does not copy the auth server's Set-Cookie headers onto the client response; surface auth-server session cookies via authSnippets (auth_request_set $sc $upstream_http_set_cookie; … add_header Set-Cookie $sc;), or use NIC's oauth2-proxy sign-in flow via authSigninURI/authSigninRedirectBasePath. tls.insecureSkipVerify maps directly to externalAuth.sslVerify: false; trustForwardHeader, headerField, and tls.certSecret (the controller never presents a client certificate to the auth server) have no equivalent, and preserveLocationHeader, preserveRequestMethod, and maxResponseBodySize likewise have no direct equivalent. authSnippets requires the controller to run with -enable-snippets.

            url: "https://errors.example.com/501.html"
        - codes: [502]
          redirect:
            code: 302
            url: "https://errors.example.com/502.html"
        - codes: [418]
          return:
            code: 404  # statusRewrites "418": 404
            type: text/plain
            body: "Not Found"
---
# errors.service — NIC cannot proxy the error page from
# another Service in-place: host the error pages behind
# their own VirtualServer and redirect to it (above)
apiVersion: k8s.nginx.org/v1
kind: VirtualServer
metadata:
  name: error-pages
spec:
  host: errors.example.com
  upstreams:
    - name: errors
      service: error-pages-svc  # service.name
      port: 80  # service.port
  routes:
    - path: /
      action:
        pass: errors

                            

                            
Note: An errorPages entry serves either a canned response (return with code, type, body, and optional headers) or a redirect (redirect with url and a code of 301, 302, 307, or 308) — there is no equivalent of errors.service that proxies the error request to a dedicated error Service in-place. Redirect URLs support a fixed set of NGINX variables (${scheme}, ${host}, ${request_uri}, ${http_x_forwarded_proto}), but none carries the response status — so Traefik's {status}/{originalStatus} placeholders are expanded at migration time with one errorPages entry per code; {url} (the escaped request URL) maps approximately to ${request_uri}. statusRewrites approximates to return.code: the client sees the rewritten status code, but return replaces the response with the inline body instead of serving the error page from a Service.

Routing (IngressRoute)

            
Translation model: a Traefik routes[].match rule decomposes into three VirtualServer constructs — Host() becomes spec.host, Path*() becomes routes[].path, and every other matcher becomes a routes[].matches[].conditions[] entry. Each condition tests exactly one of header, cookie, argument, or a whitelisted NGINX variable (11 are allowed, including $remote_addr and $request_method), and all conditions inside one match are AND-ed.

            
Route priority: the IngressRoute CRD's per-route spec.routes[].priority field — like the traefik.ingress.kubernetes.io/router.priority annotation — has no F5 NGINX Ingress Controller equivalent, because NGINX location precedence is fixed (exact > longest prefix > regex in order). To influence matching, reorder overlapping regex routes rather than assigning a numeric priority.

TCP & UDP

            
Key difference: Traefik routes raw TCP/UDP traffic with IngressRouteTCP/IngressRouteUDP resources bound to static entryPoints. The F5 NGINX Ingress Controller uses TransportServer resources bound to TCP/UDP listeners that are defined once per cluster in a GlobalConfiguration (referenced via the -global-configuration <namespace>/<name> flag). TLS & Certificates

            
Structural difference: Traefik scopes TLS settings per entry point or per router through TLSOption, and can issue certificates itself via built-in ACME (certificatesResolvers). The F5 NGINX Ingress Controller has no built-in ACME — certificate issuance moves to cert-manager — and its protocol/cipher settings are global ConfigMap keys that apply to every host. Only client-certificate authentication gains granularity: the ingressMTLS Policy attaches per application instead of per entry point.

            

                
Traefik ProxyF5 NGINX Ingress Controller

                    
                        

                            certificatesResolvers (ACME)
IngressRoute spec.tls.certResolver
router.tls.certresolver		
                            cert-manager + VirtualServer CRD spec.tls.cert-manager
                        

                        

                            

                                
Traefik Proxy
# Static configuration (traefik.yml) — built-in ACME
certificatesResolvers:
  letsencrypt:
    acme:
      email: admin@example.com
      storage: /data/acme.json  # certs live in a FILE, not in Secrets
      httpChallenge:
        entryPoint: web

# Dynamic configuration — router requests a certificate
apiVersion: traefik.io/v1alpha1
kind: IngressRoute
metadata:
  name: my-app
spec:
  entryPoints:
    - websecure
  routes:
    - match: Host(`app.example.com`)
      kind: Rule
      services:
        - name: app-service
          port: 80
  tls:
    certResolver: letsencrypt
    domains:
      - main: app.example.com

                                
F5 NGINX Ingress Controller
# 1. Install cert-manager, then create a ClusterIssuer
#    (replaces certificatesResolvers.letsencrypt)
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt
spec:
  acme:
    email: admin@example.com  # acme.email
    server: https://acme-v02.api.letsencrypt.org/directory
    privateKeySecretRef:
      name: letsencrypt-account-key  # ACME account key (was in acme.json)
    solvers:
      - http01:  # httpChallenge
          ingress:
            ingressClassName: nginx
---
# 2. Request the certificate from the VirtualServer
apiVersion: k8s.nginx.org/v1
kind: VirtualServer
metadata:
  name: my-app
spec:
  host: app.example.com  # Host(`app.example.com`)
  tls:
    secret: my-app-tls  # cert-manager will populate this Secret
    cert-manager:
      cluster-issuer: letsencrypt   # replaces certResolver "letsencrypt"
      common-name: app.example.com  # tls.domains[0].main
      issue-temp-cert: true  # serve a temporary self-signed cert until
                             # issuance completes — lets the ACME HTTP-01
                             # challenge succeed on HTTPS-only setups
  upstreams:
    - name: backend
      service: app-service
      port: 80
  routes:
    - path: /
      action:
        pass: backend

                            

                            
Certificates must be re-issued: Traefik's ACME certificates are stored in acme.json — a file, not Kubernetes Secrets — so they cannot be copied over. Plan for cert-manager to issue fresh certificates during the migration window.

                            
Note: Under tls.cert-manager, cluster-issuer and issuer are mutually exclusive and exactly one is required. Optional fields: common-name, duration and renew-before (Go durations — no d unit), usages (comma-separated), and issue-temp-cert. tls.domains[].sans (and router.tls.domains.n.sans) move to the cert-manager Certificate's dnsNames — certificate domains are a cert-manager concern, not an F5 NGINX Ingress Controller routing concern.

        enable: true         # repeat on every upstream in the split

      service: app-v2
      port: 80
      sessionCookie:
        enable: true
        name: split_session

    - path: /
      splits:
        - weight: 75  # weight 3 normalized (splits must sum to 100)
          action: { pass: app-v1 }
        - weight: 25  # weight 1 normalized (splits must sum to 100)
          action: { pass: app-v2 }

                            

                            
WRR service-selection stickiness is not preserved: Traefik's weighted.sticky.cookie pins a client to the same weighted service across requests (the WRR-level Set-Cookie records which service handled the first response). The F5 NGINX Ingress Controller selects a split per request via split_clients $request_id …, which re-rolls the weighted choice on every request, so upstreams[].sessionCookie only provides pod-level affinity within whichever upstream a request lands on — a returning client is redistributed across the weighted upstreams (e.g. a canary user will not stay on the same version). There is no direct equivalent for Traefik's WRR service-level stickiness.

                            
Note: splits[].weight takes an integer percentage (0–100) and the weights in one splits block must sum to exactly 100, with at least two splits — convert Traefik's relative weights to percentages (3:1 → 75/25). A weighted TraefikService with a single service becomes a plain action: pass. splits is also available inside matches[] for conditional splitting. weighted.sticky.cookie maps to upstreams[].sessionCookie on each upstream in the split — see Session Affinity for the full cookie field map.

EntryPoints & Networking

            
Model difference: Traefik declares every listening port as a named entryPoint in static configuration, and routers opt into them. The F5 NGINX Ingress Controller listens on 80/443 by default — extra ports are declared as GlobalConfiguration listeners (or per-Ingress via nginx.org/listen-ports), and the other entryPoint sub-settings become global ConfigMap keys in nginx-config.

            

                
Traefik ProxyF5 NGINX Ingress Controller

                    
                        

                            entryPoints.<name>.address
                            GlobalConfiguration CRD spec.listeners[]— or —nginx.org/listen-ports / nginx.org/listen-ports-ssl
                        

                        

                            

                                
                                
                            

                            

                                

                                    
Traefik Proxy
# traefik.yml (static configuration)
entryPoints:
  web:
    address: ":80"
  websecure:
    address: ":443"
  intranet:
    address: ":8080"
  syslog-udp:
    address: ":514/udp"

                                    
F5 NGINX Ingress Controller
# web/websecure = the controller's default ports 80/443 —
# nothing to declare. Custom ports become listeners:
apiVersion: k8s.nginx.org/v1
kind: GlobalConfiguration
metadata:
  name: nginx-configuration
  namespace: nginx-ingress
spec:
  listeners:
    - name: intranet-http  # entryPoints.intranet
      port: 8080           # address ":8080"
      protocol: HTTP       # TCP | UDP | HTTP
    - name: syslog-udp     # entryPoints.syslog-udp
      port: 514            # address ":514/udp"
      protocol: UDP

Certificates & ACME

            
Biggest operational delta: The F5 NGINX Ingress Controller has no built-in ACME client — cert-manager is the certificatesResolvers equivalent. Traefik stores ACME certificates in a flat file (acme.json), not in Kubernetes Secrets, so they cannot be copied over: re-issue every certificate through cert-manager before cutover.

            

                
Traefik ProxyF5 NGINX Ingress Controller

                    
                        

                            certificatesResolvers.<name>.acme.* (httpChallenge / tlsChallenge / dnsChallenge, caServer, storage)
                            cert-manager ClusterIssuer / Issuer + VirtualServer CRD spec.tls.cert-manager
                        

                        

                            

                                
Traefik Proxy
# traefik.yml — built-in ACME (Let's Encrypt)
certificatesResolvers:
  letsencrypt:
    acme:
      email: admin@example.com
      caServer: https://acme-v02.api.letsencrypt.org/directory
      storage: /data/acme.json  # certificates live in a FILE, not in Secrets
      httpChallenge:
        entryPoint: web
      # tlsChallenge: {}
      # dnsChallenge:
      #   provider: route53

# routers opt in with:
#   tls:
#     certResolver: letsencrypt

                                
F5 NGINX Ingress Controller
# 1. Install cert-manager, then create the ACME issuer
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt
spec:
  acme:
    email: admin@example.com  # acme.email
    server: https://acme-v02.api.letsencrypt.org/directory  # acme.caServer
    privateKeySecretRef:
      name: letsencrypt-account-key  # ACME account key (replaces acme.json)
    solvers:
      - http01:  # acme.httpChallenge
          ingress:
            ingressClassName: nginx
      # dns01 solvers replace acme.dnsChallenge
---
# 2. Request the certificate where the app is exposed
apiVersion: k8s.nginx.org/v1
kind: VirtualServer
metadata:
  name: my-app
spec:
  host: app.example.com
  tls:
    secret: my-app-tls  # cert-manager creates and renews this Secret
    cert-manager:
      cluster-issuer: letsencrypt  # tls.certResolver: letsencrypt
      issue-temp-cert: true  # temporary self-signed cert until the ACME
                             # order completes — keeps the HTTP-01
                             # challenge working on an HTTPS-only host
  upstreams:
    - name: backend
      service: backend-svc
      port: 80
  routes:
    - path: /
      action:
        pass: backend

                            

                            
Note: spec.tls.cert-manager at v5.5.1 also accepts issuer (a namespaced Issuer — mutually exclusive with cluster-issuer), issuer-kind / issuer-group (out-of-tree issuers), common-name, duration, renew-before (Go durations, no d unit), and usages. If you are staying on Ingress resources rather than VirtualServer, use cert-manager's own cert-manager.io/cluster-issuer annotation on the Ingress instead.
