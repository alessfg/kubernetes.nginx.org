NGINX Traefik Migration Tool

Under construction: This is a scaffold for the Traefik → F5 NGINX Ingress Controller migration tool. Content is being added incrementally — see the repository for progress.
Version Reference: This tool is based on Traefik v3.7.1 and F5 NGINX Ingress Controller v5.5.0.

Overview

This interactive guide helps you migrate from Traefik Proxy (traefik/traefik) to the F5 NGINX Ingress Controller (nginx/kubernetes-ingress). It covers Traefik annotations on standard Ingress resources, the full set of OSS HTTP and TCP middlewares, IngressRoute / IngressRouteTCP / TraefikService translations, and Traefik static-config to NIC ConfigMap mappings — across three tabs:

  • Getting Started — Why migrate, CRD overviews, installation, and a Traefik-aware migration checklist
  • Traefik Config Analyzer — (coming soon) Paste an IngressRoute, Middleware, or annotated Ingress and get migration suggestions with ready-to-use NIC YAML
  • Reference Guide — Complete annotation, middleware, and CRD mapping tables with side-by-side examples

Why Migrate?

Traefik is actively maintained, so the case for switching is feature- and ecosystem-driven rather than urgency-driven. The F5 NGINX Ingress Controller (NIC) is the right target when one or more of the following applies:

  • NGINX Plus features — JWT validation, OIDC integration, F5 WAF (App Protect), API Key auth, native session persistence with sticky cookies, and active backend health checks are first-class Policy CRDs and annotations in NIC.
  • F5 commercial support — Paid 24×7 support, FIPS 140-3 builds, and access to F5's broader security stack (App Protect WAF, App Protect DoS) for regulated workloads.
  • NGINX ecosystem alignment — If your team already runs NGINX OSS or NGINX Plus outside Kubernetes, NIC keeps the configuration model and operational tooling consistent. Snippet annotations let you inject raw NGINX directives when CRDs don't cover a niche use case.
  • Battle-tested data plane — NIC ships NGINX as its data plane (OSS or Plus). NGINX has decades of production hardening as an HTTP server and reverse proxy.
  • Schema-validated CRDs — VirtualServer, VirtualServerRoute, Policy, TransportServer, and GlobalConfiguration provide structured, validated configuration with IDE autocompletion — closely matching Traefik's own CRD-native philosophy.

Migration friction worth flagging up front (covered in detail throughout the Reference Guide):

  • Traefik's Chain middleware has no NIC equivalent — chains are flattened during migration; the named-chain abstraction is lost.
  • The Traefik match-string DSL (Host(`...`) && PathPrefix(`...`)) translates to VirtualServer routes[].path + routes[].matches[], but matchers like HostRegexp, Query, and ClientIP require careful per-case mapping.
  • Cert resolvers (ACME) are not built into NIC. Plan to install cert-manager for certificate issuance and reference the issued Secret in spec.tls.secretName.
  • Plugins loaded via Traefik's plugin catalog are not migratable — each needs a manual review for an NGINX module or snippet equivalent.
  • EntryPoints beyond web/websecure may not have a clean NIC equivalent — HTTP listening ports are configured via NIC CLI flags / Helm values, not per-Ingress.
  • TraefikService mirroring, per-route metrics / tracing toggles, router.priority, service.nodeportlb, and GrpcWeb translation have no NIC equivalent — documented workarounds (or "keep Traefik for these specific routes") apply.

Phased Migration Strategy

Rather than performing a "big bang" cutover, F5 recommends a phased migration that lets you validate each workload before decommissioning Traefik. Both controllers can run side-by-side in the same cluster using separate IngressClass resources.

  1. Deploy side-by-side — Install the F5 NGINX Ingress Controller alongside Traefik using a distinct IngressClass (e.g., nginx-nic). Traefik continues to serve all existing traffic.
  2. Migrate workload-by-workload — For each application:
    • If the workload uses standard Ingress + Traefik annotations, translate annotations per the Reference Guide, then update ingressClassName to the NIC class.
    • If it uses IngressRoute / Middleware CRDs, generate equivalent VirtualServer + Policy CRDs, apply them, then delete the IngressRoute. The TraefikService weighted-services pattern usually maps to VirtualServer splits.
    • Validate routing, TLS, and any middleware-driven behavior (auth, rate limiting) in staging before promoting.
  3. Migrate edge concerns — Cert issuance (ACME → cert-manager), entry-point overrides, and any plugin-driven behavior need a separate path (see the "Why Migrate" caveats above).
  4. Decommission Traefik — After all workloads have moved and been validated, remove Traefik's deployment, its IngressClass, and the Traefik CRDs (kubectl delete crd ingressroutes.traefik.io ingressroutetcps.traefik.io middlewares.traefik.io ...).
Tip: The Traefik Config Analyzer tab (coming soon) will accept individual IngressRoute / Middleware / annotated Ingress manifests and emit the equivalent NIC configuration — use it during step 2 to accelerate per-workload conversion.

CRDs Summary

The F5 NGINX Ingress Controller uses Custom Resource Definitions (CRDs) to extend Kubernetes with NGINX-specific configuration. CRDs provide type-safe, validated configuration that goes beyond what standard Ingress annotations can offer.

VirtualServer CRD

The primary CRD for configuring HTTP/HTTPS load balancing. The migration target for Traefik IngressRoute. Provides advanced traffic management capabilities with full validation.

  • Traffic Splitting: Route percentages of traffic to different backends — the migration target for Traefik TraefikService weighted services
  • Content-Based Routing: Route based on headers, cookies, arguments, or variables — maps to Traefik's Headers / Query matchers
  • Path Rewriting: Modify request URIs — replaces Traefik StripPrefix, ReplacePath, ReplacePathRegex middlewares
  • Error Pages: Custom responses — replaces Traefik Errors middleware
  • Request/Response Headers: Add, modify, or remove headers — replaces parts of Traefik Headers middleware
  • Health Checks: Active health monitoring (NIC Plus) — closest analog to Traefik CircuitBreaker
  • Session Persistence: Cookie-based sticky sessions (NIC Plus) — replaces Traefik service.sticky.cookie.* annotations

VirtualServerRoute CRD

Enables delegation of route configuration to different namespaces. Useful when migrating Traefik setups where teams managed their own IngressRoutes.

  • Cross-Namespace Routing: Reference services in different namespaces
  • Path Delegation: Delegate path prefixes to other teams/namespaces
  • Independent Updates: Teams update routes without modifying the parent VirtualServer
  • Access Control: Limit which namespaces can define routes for a host

Policy CRD

Reusable security and traffic policies attached to VirtualServers or specific routes. The migration target for most Traefik HTTP Middleware kinds.

  • Rate Limiting: Token-bucket per source — replaces Traefik RateLimit middleware
  • Access Control: IP allowlists/denylists — replaces Traefik IPAllowList middleware
  • Basic Auth: htpasswd-based authentication (OSS) — replaces Traefik BasicAuth middleware
  • External Auth: Subrequest-based auth (new in v5.5, OSS) — replaces Traefik ForwardAuth middleware
  • JWT Validation: JSON Web Tokens (Plus) — replaces Traefik Hub JWT middleware
  • OIDC: Native OpenID Connect (Plus) — replaces Traefik Hub OIDC middleware
  • API Key Auth: Header/query-based keys (Plus) — replaces Traefik Hub APIKey middleware
  • mTLS: Client cert verification (ingress) and backend TLS (egress) — replaces Traefik PassTLSClientCert + ServersTransport mTLS
  • WAF: F5 WAF for NGINX (Plus) — replaces Traefik Hub WAF middleware

TransportServer CRD

Configures TCP/UDP load balancing for non-HTTP protocols. The migration target for Traefik IngressRouteTCP and IngressRouteUDP.

  • TCP Load Balancing: MySQL, PostgreSQL, Redis, custom TCP services — replaces IngressRouteTCP
  • UDP Load Balancing: DNS, RADIUS, gaming protocols — replaces IngressRouteUDP
  • TLS Passthrough: SNI-based routing without termination — replaces IngressRouteTCP with tls.passthrough: true
  • Health Checks: TCP/UDP health monitoring
  • Session Persistence: IP hash-based session affinity

GlobalConfiguration CRD

Cluster-wide configuration for the F5 NGINX Ingress Controller. Defines listener ports that TransportServer resources reference — the closest analog to Traefik EntryPoints.

  • TCP/UDP Listeners: Define ports for non-HTTP traffic — replaces Traefik EntryPoints for TCP/UDP
  • TLS Passthrough Listeners: Configure SNI-based routing ports
  • Global Settings: Controller-wide configuration options

Installing CRDs

CRDs can be installed via Helm or manually using manifests, depending on your deployment method. See the F5 NGINX Ingress Controller installation docs for full instructions.

Helm (example)

If you install the F5 NGINX Ingress Controller via Helm, CRDs are included automatically. To ensure they are enabled:

helm install nginx-ingress oci://ghcr.io/nginx/charts/nginx-ingress --version 2.5.0 --set controller.enableCRDs=true

The controller.enableCRDs flag is true by default. See the Helm installation guide for details.

Manifests (example)

If you deploy using manifests, install CRDs manually before deploying the controller:

kubectl apply -f https://raw.githubusercontent.com/nginx/kubernetes-ingress/v5.5.0/deploy/crds.yaml

Replace the version number with your target controller version. See the manifest installation guide for the full step-by-step process.

Note: The crds.yaml file contains all CRDs (VirtualServer, VirtualServerRoute, Policy, TransportServer, GlobalConfiguration) in a single manifest.
Tip: If you were using Traefik's built-in ACME client for cert issuance, install cert-manager before cutting traffic over — NIC delegates certificate issuance to cert-manager and references the issued Secret in spec.tls.secretName.

Migration Checklist

  • Install NIC CRDs via Helm (controller.enableCRDs=true) or manifests (deploy/crds.yaml)
  • Install cert-manager if Traefik was issuing certificates via ACME (NIC has no built-in ACME client)
  • Deploy NIC side-by-side with Traefik using a distinct IngressClass (e.g., nginx-nic)
  • For workloads on standard Ingress + Traefik annotations: translate router.* and service.* annotations to NIC equivalents (see Reference Guide)
  • For workloads using IngressRoute: rewrite to VirtualServer with attached Policy CRDs
  • For workloads using IngressRouteTCP: rewrite to TransportServer (with optional GlobalConfiguration listener)
  • For workloads using TraefikService weighted routing: convert to VirtualServer splits
  • TraefikService mirroring: no direct equivalent — use location-snippets with the NGINX mirror directive, or accept loss
  • Flatten any Chain middleware: inline each chained middleware into the consuming VirtualServer or Policy attachment
  • Convert RateLimit middleware to Policy CRD rateLimit
  • Convert BasicAuth middleware to Policy CRD basicAuth (OSS) or nginx.org/basic-auth-secret annotation
  • Convert IPAllowList middleware to Policy CRD accessControl
  • Convert ForwardAuth middleware to Policy CRD externalAuth (new in NIC v5.5, OSS-compatible)
  • Convert Headers middleware to VirtualServer requestHeaders/responseHeaders or nginx.org/proxy-set-headers
  • Convert StripPrefix / ReplacePath / AddPrefix middlewares to nginx.org/rewrites
  • Convert RedirectScheme middleware to nginx.org/redirect-to-https
  • Convert Compress middleware to global gzip ConfigMap keys (NIC has no per-route compression toggle)
  • Convert Buffering middleware to nginx.org/proxy-* annotations or ConfigMap proxy-* keys
  • Convert HSTS via Headers middleware to NIC ConfigMap hsts* keys
  • Replace Traefik certResolver TLS with cert-manager-issued Secrets referenced from spec.tls.secretName
  • Accept that per-route metrics/tracing toggles have no NIC equivalent (global only) — adjust observability expectations
  • For gRPC-Web routes: NIC has no translation — either keep Traefik or move translation to backend (Envoy sidecar, server-side library)
  • Test routing and middleware behavior in staging
  • Update monitoring and alerting (NIC exports its own Prometheus metric set)
